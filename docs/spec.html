Project Spec

gmeta: Structured metadata for Git-native workflows

gmeta is a Rust CLI that lets teams attach durable, queryable metadata to Git entities like commits, branches, file paths, change IDs, or the project itself.

It stores metadata in local SQLite for fast read/write operations, and exchanges state over Git refs so metadata can be exchanged and become eventually consistent with normal Git tooling.

The Point Of This Project

Git stores source history well, but many modern workflows need extra machine-readable context: model/version provenance, review state, agent transcripts, audit notes, or task metadata. gmeta makes this data first-class without inventing a separate service.

Problem Teams need metadata bound to Git objects, searchable locally, and shareable across remotes.
Approach Store mutable state in SQLite, keep a mutation log, and serialize current state to Git refs for exchange.
Result Fast local CRUD with distributed exchange that works with existing Git hosting.
Design Principle Keep exchange format simple and diffable by using normal Git trees and blobs.

Flow Steps

    Data Model

    Each metadata value is identified by target + key and stores value + value_type.

    So, for example, if you wanted to store the model of an agent used to generate a commit at it's sha, you might have:

    Target commit:620b1bcba8657e7f1dfdbf0c1170b5e309f15575
    Key agent:model
    Value Type string
    Value opus-4.6

    Target formats

    These are the formats currently supported, but new formats should be relatively simple to add in the future (ie, 'byte' level attestations, etc).

    Type Format Notes
    commitcommit:<sha>Partial SHAs are resolved to full 40 chars.
    change-idchange-id:<id>Arbitrary ID, typically UUID-like.
    branchbranch:<name-or-id>Branch UUID or name.
    pathpath:<file/path>Project file path.
    projectprojectGlobal namespace, no target sub ID.

    Value types

    Type Stored form Behavior
    stringJSON string in SQLiteSimple set/get/remove.
    listlist_values rows keyed by metadata row id (value, timestamp)Append/pop mutate rows directly; deterministic Git entry names on serialize.

    CLI Surface

    Primary commands exposed by this reference implementation. There are two different sets of commands, the metadata CRUD primatives and the exchage commands.

    Metadata commands

    These commands are to set, get and remove metadata on targets. The data can either be a single string/blob or a list of values.

    gmeta set [-t string|list] [-F file] <target> <key> [value]
gmeta get [--json] [--with-authorship] <target> [key]
gmeta rm <target> <key>
gmeta list:push <target> <key> <value>
gmeta list:pop <target> <key> <value>

    Exchange commands

    There are two exchange commands.

    gmeta serialize
gmeta materialize [remote]

    One to serialize the current state of metadata values into a Git reference (commit/tree) (stored by default under `refs/meta/local`) so they can be pushed to any Git remote with normal Git tooling.

    Another to materialize data fetched from a remote that another user pushed to incorporate any changes they made into your local database. This data is stored under `refs/meta/[remote-name]` so you could theoretically have several of them.

    1
    Set/get metadata quickly
    Backed by SQLite at .git/gmeta.sqlite.
    2
    Serialize for exchange
    Writes metadata snapshot to refs/<namespace>/local (namespace from meta.namespace, default meta).
    3
    Materialize from remotes
    Imports refs under refs/<namespace>/..., merges into local meta ref, and updates SQLite.

    Git Exchange Format

    Metadata is serialized into Git trees so it is transparent to inspect, diff, transfer, and merge.

    Tree path shape

    Base:
  project target:    project
  other targets:     {type}/{first_2_chars}/{last_3_chars}/{full_target}

Key mapping:
  key root dir:       k/
  key segments:       ":"-split, stored as raw directory names
  string terminal:    __value
  list terminal dir:  __list

Example key:
  agent:model -> k/agent/model/__value

    The key trie + reserved terminals prevents Git blob/directory collisions for keys like agent and agent:model.

    The target tree format ({type}/{first-2}/{last-3}/{full}) provides better fanout and distribution so trees don't get too large with lots of targets.

    Key validation rules: non-empty key, non-empty segments, no . or .., no / or null bytes, and no reserved segments k, __value, __list.

    The k/ boundary also reserves target-level keyspace for future metadata files such as __format/__schema, __merge_policy, __tombstones/, and __signature/__digest.

    String example

    commit/13/0ae/13a7...80ae/k/agent/model/__value -> "claude-4.6"

    List example

    branch/sc/eef/sc-branch-1-deadbeef/k/agent/chat/__list/1771232450203-23c0f
branch/sc/eef/sc-branch-1-deadbeef/k/agent/chat/__list/1771232450204-0d5f2

    List entry names are {timestamp_ms}-{first_5_sha256(value)}. This is so the trees are almost always easily mergeable if multiple users add to the same list asynchronously.

    Removal tombstone example

    commit/13/0ae/13a7...80ae/__tombstones/k/agent/model/__deleted
  -> {"timestamp": 1771232450000, "email": "user@example.com"}

    Removes are persisted in SQLite tombstones and serialized for exchange; re-setting a key clears its tombstone.

    Materialize + Merge Behavior

    Materialization reads remote metadata refs, then either fast-forwards or performs a three-way merge.

    If local and remote metadata refs have no common ancestor, materialize switches to a two-way merge and remote state wins overlapping key conflicts.

    Fast-forward path If local meta ref is ancestor of remote ref, update local ref directly, apply tombstones, and remove keys missing from the incoming value tree.
    Three-way merge path Build base, local, remote key maps and resolve per key.
    String conflict rule Later commit timestamp wins (ties favor local).
    List conflict rule Union list entries by entry name; preserve sorted order by timestamp-hash.
    Removal conflict rule Tombstones remove keys, but if a value is concurrently modified, the surviving merged value wins over tombstone.
    No common ancestor rule Use two-way merge and prefer remote state for overlapping keys (value-vs-value and value-vs-tombstone).

    Putting this in Production

    This is a spec and example plumbing implementation, there is a fair amount of UI and porcelain integration needed to make the system truly useable.

    • Setup/Config of Meta Remote Push/Fetch spec
    • Auto serialize pre-push, push to meta tracking branch(es) too
    • If push is a non-fast forward, re-fetch, merge, materialize, re-push
    • Auto materialize post-fetch, fetch from meta trancking simultaneously
    • Show metadata with `but show`, deal with larger blobs (transcripts, etc) or binaries (signatures, etc)
