gmeta — Fanout Scheme Analysis

git tree sharding strategy for commit metadata storage

Background

gmeta stores metadata for git objects (commits, branches, paths, …) as blobs inside git trees, serialized under refs/meta/local. Each target type needs a sharding strategy so that no single tree node holds too many entries as the repo grows.

Three candidate schemes were benchmarked for commit-SHA targets at multiple scales: 10k, 100k, and 1M objects.

Schemes tested

scheme buckets
{first2} 256
{first3} 4 096
{first2}/{next2} 65 536

Write, read, diff, and pack-size were measured for each scheme using the gmeta fanout-bench command on synthetic SHA targets with uniform distribution.

Benchmark runs

Key findings

!
Diff cost explodes with depth. At 1M objects, {first2}/{next2} is 42× slower to diff than {first2} (30.5 s vs 0.7 s) because diff_tree_to_tree must open every intermediate tree node on the path to each change.
i
Pack size is nearly identical across all three schemes at 1M objects (106–116 MB). Blob content dominates; the extra tree objects from deeper sharding compress away.
i
Read speed favours {first2}/{next2} slightly (600 µs vs 1332 µs per object at 1M) because tree.get_path() descends into smaller leaf buckets. The difference is small compared to the diff penalty.
✓
256 buckets is sufficient for commit SHAs. At 10M commits each {first2} bucket holds ~39k entries — well within git's efficient range. A second level would only be needed beyond ~500M commits with metadata.

Decisions

target scheme
commit
{first2}/{sha}
SHA is uniform. One prefix level handles hundreds of millions of commits with fast diffs.
change-id
{first2}/{id}
Change IDs are opaque hex — already uniform, same logic as commit.
branch
{first2(hash(name))}/{name}
Branch names cluster on prefixes (feature/, fix/, …). Hashing before taking first2 gives uniform distribution.
path
{path}
The path itself is hierarchical. Git's tree structure provides natural directory-level fanout for free.
project
project
Singleton — no sharding needed.
